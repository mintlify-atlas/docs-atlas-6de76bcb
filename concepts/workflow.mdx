## Stage 1: Graph Construction

<Info>
  This stage builds a knowledge graph from seed materials using Zep Cloud's GraphRAG capabilities.
</Info>

### Input Requirements

Users provide two key inputs:

1. **Seed Materials** (PDF, Markdown, or TXT files up to 50MB)
   - News articles
   - Policy documents  
   - Social media posts
   - Any text describing the scenario to simulate

2. **Simulation Requirement** (natural language description)
   - What outcome to predict
   - What variables to inject
   - What scenarios to explore

### Ontology Generation

Before graph construction, an LLM generates a domain-specific ontology:

```python
# From ontology_generator.py
class OntologyGenerator:
    def generate(
        self,
        document_texts: List[str],
        simulation_requirement: str,
        additional_context: Optional[str] = None
    ) -> Dict[str, Any]:
        # LLM analyzes content and generates ontology
        messages = [
            {"role": "system", "content": ONTOLOGY_SYSTEM_PROMPT},
            {"role": "user", "content": user_message}
        ]
        
        result = self.llm_client.chat_json(
            messages=messages,
            temperature=0.3,
            max_tokens=4096
        )
        
        # Returns entity_types and edge_types
        return result
```

**Ontology structure:**
- **10 entity types** (8 specific + 2 generic fallbacks: Person, Organization)
- **6-10 edge types** (relationships like WORKS_FOR, COMMENTS_ON, RESPONDS_TO)
- **Attributes** for each type (avoiding reserved names: uuid, name, summary)

Example ontology output:

```json

      "description": "University student involved in the event",

        {"name": "department", "type": "text"},

      ]
    },
    {
      "name": "University",

```

### Graph Building Process

With the ontology defined, MiroFish constructs the graph:

<Steps>
  <Step title="Create Graph">
    A new Zep Cloud graph is created with a unique ID: `mirofish_{random_hash}`
  </Step>
  
  <Step title="Set Ontology">
    The generated ontology is registered with Zep using dynamic Pydantic models:
    
    ```python
    # Zep requires Pydantic models for ontology
    entity_class = type(name, (EntityModel,), {
        "__doc__": description,
        "attribute_name": Field(description=attr_desc, default=None)
    })
    
    self.client.graph.set_ontology(
        graph_ids=[graph_id],
        entities=entity_types,
        edges=edge_definitions
    )
    ```
  </Step>
  
  <Step title="Split Text">
    Documents are chunked into 500-character segments with 50-character overlap for processing.
  </Step>
  
  <Step title="Batch Processing">
    Text chunks are sent to Zep in batches of 3:
    
    ```python
    episodes = [
        EpisodeData(data=chunk, type="text")
        for chunk in batch_chunks
    ]
    
    batch_result = self.client.graph.add_batch(
        graph_id=graph_id,
        episodes=episodes
    )
    ```
  </Step>
  
  <Step title="Wait for Processing">
    Zep processes episodes asynchronously. MiroFish polls each episode's `processed` status until complete (timeout: 600 seconds).
  </Step>
  
  <Step title="Retrieve Graph Data">
    Final graph statistics and entity/edge data are fetched:
    
    ```python
    nodes = fetch_all_nodes(self.client, graph_id)  # Paginated
    edges = fetch_all_edges(self.client, graph_id)  # Paginated
    
    return GraphInfo(
        graph_id=graph_id,
        node_count=len(nodes),
        edge_count=len(edges),
        entity_types=list(entity_types)
    )
    ```
  </Step>
</Steps>

**Typical output:**
- **Nodes**: 50-200 entities extracted from documents
- **Edges**: 100-500 relationships between entities
- **Processing time**: 2-5 minutes for ~10K words

## Stage 2: Environment Setup

<Info>
  This stage transforms graph entities into simulation agents with detailed personalities and social media behavior profiles.

### Agent Profile Generation

Each entity in the graph becomes an autonomous agent:

```python

        )
```

### Context Enrichment

Before profile generation, MiroFish enriches entity context using Zep's hybrid search:

1. **Search edges** (relationships/facts) related to the entity
2. **Search nodes** (related entities) to understand connections
3. **Extract attributes** from the entity itself
4. **Combine into rich context** (~3000 characters max)

```python
def _build_entity_context(self, entity: EntityNode) -> str:
    context_parts = []
    
    # 1. Entity attributes
    if entity.attributes:
        context_parts.append("### Entity Attributes\n" + format_attrs(entity.attributes))
    
    # 2. Related edges (facts)
    if entity.related_edges:
        context_parts.append("### Related Facts\n" + format_edges(entity.related_edges))
    
    # 3. Related nodes
    if entity.related_nodes:
        context_parts.append("### Related Entities\n" + format_nodes(entity.related_nodes))
    
    # 4. Zep hybrid search for additional context
    zep_results = self._search_zep_for_entity(entity)
    if zep_results.get("facts"):
        context_parts.append("### Additional Facts\n" + format_facts(zep_results["facts"]))
    
    return "\n\n".join(context_parts)
```

### Profile Details

The LLM generates comprehensive profiles with:

- **200-word bio** (public-facing description)
- **2000-word persona** (internal behavior guide) including:
  - Basic demographics (age, gender, location, profession)
  - Background and relationship to the event
  - Personality traits and MBTI type
  - Social media behavior patterns (posting frequency, interaction style)
  - Stances and attitudes on key topics
  - Unique characteristics and speech patterns
  - **Individual memory** of their role in the event

**Platform-specific formats:**
- **Twitter**: CSV format with columns `user_id,name,username,user_char,description`
- **Reddit**: JSON format with user_id, username, persona, age, gender, MBTI, karma

### Parallel Processing

Profile generation is parallelized with a thread pool:

```python
with concurrent.futures.ThreadPoolExecutor(max_workers=5) as executor:
    futures = {
        executor.submit(generate_single_profile, idx, entity): (idx, entity)
        for idx, entity in enumerate(entities)
    }
    
    for future in concurrent.futures.as_completed(futures):
        result_idx, profile, error = future.result()
        profiles[result_idx] = profile
        
        # Real-time file writing
        save_profiles_realtime()
```

**Performance:**
- **5 parallel workers** generate profiles concurrently
- **~10-20 seconds per profile** depending on LLM speed
- **Realtime file updates** allow simulation to start before all profiles complete

### Real-Time Monitoring

A background thread monitors both action logs:

```python
def _monitor_simulation(simulation_id: str):
    twitter_position = 0
    reddit_position = 0
    
    while process.poll() is None:  # Process still running
        # Read new lines from Twitter log

            twitter_actions_log, twitter_position, state, "twitter"
```

### Time Simulation

MiroFish simulates time passage with configurable granularity:

```python

```

**Timeline:**
- Default simulation: **72 hours** (3 days)
- Each round: **60 minutes** of simulated time
- Total rounds: **72 rounds**
- Agent activity varies by time of day (realistic diurnal patterns)

## Stage 4: Report Generation

<Info>
  The Report Agent uses ReACT (Reasoning + Acting) to generate comprehensive prediction reports.
</Info>

### Two-Phase Generation

**Phase 1: Outline Planning**

The agent first plans the report structure:

```python
# From report_agent.py
def plan_outline(self) -> ReportOutline:
    # Get simulation context
    context = self.zep_tools.get_simulation_context(
        graph_id=self.graph_id,
        simulation_requirement=self.simulation_requirement
    )
    
    # LLM generates 2-5 section outline
    response = self.llm.chat_json(
        messages=[
            {"role": "system", "content": PLAN_SYSTEM_PROMPT},
            {"role": "user", "content": format_context(context)}
        ],
        temperature=0.3
    )
    
    return ReportOutline(
        title=response["title"],
        summary=response["summary"],
        sections=[ReportSection(title=s["title"]) for s in response["sections"]]
    )
```

**Phase 2: Section Generation with ReACT**

Each section is generated using a ReACT loop:

<Steps>
  <Step title="Thought">
    Agent thinks about what information is needed for this section.
  </Step>
  
  <Step title="Action">
    Agent calls a tool (insight_forge, panorama_search, quick_search, or interview_agents).
    
    ```xml
    <tool_call>
    {"name": "insight_forge", "parameters": {"query": "How did students react?"}}
    </tool_call>
    ```
  </Step>
  
  <Step title="Observation">
    System executes tool and injects results back to agent.
    
    ```python
    observation_msg = f"""
    Observation (Tool Result):
    
    === Tool {tool_name} returned ===
    {result}
    
    Already called {tool_calls_count}/{max_tool_calls} tools.
    - If sufficient info: Output "Final Answer:" + section content
    - If need more info: Call another tool
    """
    ```
  </Step>
  
  <Step title="Repeat or Answer">
    Agent repeats Thought → Action → Observation up to 5 times per section, then outputs final content.
  </Step>
</Steps>

### Available Tools

The Report Agent has 4 powerful tools:

<CardGroup cols={2}>
  <Card title="insight_forge" icon="brain">
    **Deep analytical search**
    
    Auto-decomposes query into sub-questions, retrieves multi-dimensional facts, and synthesizes insights.
  </Card>
  
  <Card title="panorama_search" icon="globe">
    **Breadth-first search**
    
    Gets complete timeline of event evolution, including expired/historical facts vs. current facts.
  </Card>
  
  <Card title="quick_search" icon="search">
    **Fast fact lookup**
    
    Semantic search for specific information validation.
  </Card>
  
  <Card title="interview_agents" icon="microphone">
    **Real agent interviews**
    
    Calls OASIS API to interview actual simulation agents, gets first-person perspectives from different roles.
  </Card>
</CardGroup>

### Report Output

Final report is generated as Markdown with:
- **Title and executive summary**
- **2-5 main sections** (no sub-sections)
- **Rich quotations** from agent actions and interviews
- **Evidence-based predictions** grounded in simulation results
- **Bilingual support** (auto-translates English tool results to match report language)

**Logging:**
- `agent_log.jsonl`: Every tool call, LLM response, and thought process
- `console_log.txt`: Human-readable progress log
- `report.md`: Final Markdown report

## Performance Considerations

<Warning>
  Simulation is LLM-intensive. A 72-hour simulation with 100 agents can make 10,000+ LLM calls.
</Warning>

**Resource usage:**
- **Graph construction**: ~2-5 minutes for 10K words
- **Profile generation**: ~10-30 minutes for 100 agents (parallel)
- **Simulation**: ~30-120 minutes depending on rounds and agent count
- **Report generation**: ~5-15 minutes for 3-5 sections

**Optimization tips:**
- Use smaller `max_rounds` for testing (e.g., 10 rounds = 10 hours simulated)
- Reduce agent count by filtering low-importance entities
- Use faster LLM models (Qwen-Plus recommended over GPT-4)
- Enable parallel platform simulation for comprehensive social media coverage

**Cost estimates** (using Qwen-Plus):
- Graph construction: $0.50-2.00
- Profile generation (100 agents): $5-15
- Simulation (72 rounds, 100 agents): $50-150
- Report generation: $2-5
- **Total for full run**: $60-175
