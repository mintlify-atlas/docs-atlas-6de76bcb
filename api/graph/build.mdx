## Build Knowledge Graph

<Note>
  Build a knowledge graph from extracted text using the generated ontology. This is an asynchronous operation that returns a task_id for progress tracking.
</Note>

```bash
POST /api/graph/build
```

### Request Body (JSON)

<ParamField body="project_id" type="string" required>
  Project ID from ontology generation (e.g., `proj_abc123`)
</ParamField>

<ParamField body="graph_name" type="string">
  Custom name for the knowledge graph (defaults to project name)
</ParamField>

<ParamField body="chunk_size" type="integer" default="500">
  Text chunk size for graph building (characters)
</ParamField>

<ParamField body="chunk_overlap" type="integer" default="50">
  Overlap between consecutive chunks (characters)
</ParamField>

<ParamField body="force" type="boolean" default="false">
  Force rebuild if graph is already being built or completed
</ParamField>

### Response

<ResponseField name="success" type="boolean">
  Whether the request succeeded
</ResponseField>

<ResponseField name="data" type="object">
  <Expandable title="Build task information">
    <ResponseField name="project_id" type="string">
      Project identifier
    </ResponseField>
    <ResponseField name="task_id" type="string">
      Task ID for progress tracking
    </ResponseField>
    <ResponseField name="message" type="string">
      Instructions for checking progress
    </ResponseField>
  </Expandable>
</ResponseField>

### Example

<CodeGroup>

## Check Build Progress

<Note>
  Use the task_id to monitor graph building progress.
</Note>

```bash

```

### Path Parameters

<ParamField path="task_id" type="string" required>

</ParamField>

<ResponseField name="success" type="boolean">
  Whether the request succeeded
</ResponseField>

<ResponseField name="data" type="object">
  <Expandable title="Task status">
    <ResponseField name="task_id" type="string">
      Task identifier
    </ResponseField>
    <ResponseField name="status" type="string">

## Graph Building Process

The graph building workflow:

1. **Initialize** (0-10%): Set up Zep graph and validate ontology
2. **Text Chunking** (10-15%): Split documents into processable chunks
3. **Add Text Batches** (15-55%): Upload text chunks to Zep in batches
4. **Process Episodes** (55-90%): Wait for Zep to extract entities and relationships
5. **Retrieve Data** (90-100%): Fetch final graph statistics

## Configuration Parameters

### Chunk Size

<ParamField body="chunk_size" type="integer" default="500">
  Recommended values:
  - **Small documents (< 10 pages):** 300-500 characters
  - **Medium documents (10-50 pages):** 500-800 characters
  - **Large documents (> 50 pages):** 800-1000 characters
</ParamField>

### Chunk Overlap

<ParamField body="chunk_overlap" type="integer" default="50">
  Overlap ensures context continuity:
  - **Minimum:** 20 characters
  - **Recommended:** 50-100 characters (10-15% of chunk_size)
  - **Maximum:** 200 characters
</ParamField>

## Force Rebuild

To rebuild an existing graph:

```bash
curl -X POST http://localhost:5001/api/graph/build \
  -H "Content-Type: application/json" \
  -d '{
    "project_id": "proj_abc123",
    "force": true
  }'
```

<Warning>
  Force rebuilding will reset project status and create a new graph. Previous graph data will be lost.
</Warning>
